# Руководство поддержки и устранения неполадок — VITBON ККМ

## Назначение

Документ для техподдержки и старших смен по диагностике типовых проблем:
- вход и лицензия;
- синхронизация чеков и каталога;
- статусы модулей (интернет/облако/ОФД/ЕГАИС/ЧЗ);
- восстановление работоспособности без потери кассовых данных.

## Быстрый triage (5 минут)

1. Зафиксируйте контекст:
   - устройство и версия приложения;
   - время проблемы;
   - действие пользователя перед сбоем.
2. Проверьте экран «Статусы»:
   - Интернет;
   - Облачный сервер;
   - ОФД;
   - Лицензия;
   - ЕГАИС/ЧЗ (если модули активны).
3. Проверьте критичный признак блокировки:
   - есть ли экран блокировки лицензии;
   - есть ли предупреждение о локальном входе.
4. Если продажа невозможна — сначала восстановите смену/лицензию, потом повторите действие.

## Диагностика по симптомам

### 1) Не удаётся войти

Проверки:
- корректность PIN;
- наличие кассира в локальной базе;
- доступность backend (если ожидается online-валидация).

Что делать:
- при offline сценарии выполнить локальный вход;
- если backend недоступен, убедиться, что отображается non-blocking предупреждение, а сессия не сбрасывается;
- при массовом отказе входа эскалировать инцидент на backend/auth.

### 2) Экран «Лицензия неактивна»

Проверки:
- доступность `/api/v1/license/check`;
- корректность `deviceId`;
- локальные метки времени последней успешной проверки.

Что делать:
- восстановить сеть и повторить проверку лицензии;
- при подтверждённой просрочке — передать в коммерческий/лицензионный контур;
- при расхождении статуса между устройствами — собрать логи и эскалировать.

### 3) Чеки не синхронизируются

Проверки:
- очередь `PENDING_SYNC` не растёт ли бесконтрольно;
- есть ли ответы `200` на `POST /api/v1/checks/sync`;
- переходят ли чеки в `SYNCED`, заполняется ли `syncedAt`.

Что делать:
- проверить сеть и endpoint backend;
- принудительно запустить повтор синхронизации;
- при постоянных ошибках собрать payload/response и передать в backend.

### 4) В отчётах «не те цифры»

Проверки:
- выбранный период (смена/день/неделя/месяц);
- источник данных на экране отчётов (`Источник: Облако` / `Источник: Локально`);
- наличие синхронизированных чеков в выбранном интервале.

Что делать:
- если источник локальный, проверить локальные чеки за период;
- если источник облако, проверить доступность `/api/v1/reports`;
- при расхождении источников зафиксировать скрин и значения, эскалировать в reports lane.

### 5) Проблемы ЕГАИС/Честный ЗНАК

Проверки:
- активирован ли модуль фичефлагом;
- доступны ли интеграционные endpoint-ы:
  - `/api/v1/egais/incoming`
  - `/api/v1/egais/tara`
  - `/api/v1/chaseznak/sell`
  - `/api/v1/chaseznak/verify-age`

Что делать:
- при недоступности модуля — отключить пользовательский сценарий и сообщить причину кассиру;
- при массовой ошибке валидации кодов — собрать примеры кодов/ответов и эскалировать.

## Набор контрольных API для поддержки

- `POST /api/v1/auth/login` — проверка PIN backend-валидации.
- `GET /api/v1/statuses` — оперативные статусы интеграций.
- `POST /api/v1/checks/sync` — подтверждение прохождения синка чеков.
- `GET /api/v1/reports` и `GET /api/v1/reports/sales` — сверка агрегатов продаж.
- `GET /api/v1/reports/movement` — сверка движения товаров.
- `POST /api/v1/license/check` — проверка лицензионного статуса.

Полный контракт API: `backend/src/main/resources/api/openapi.yaml`.

## Когда эскалировать сразу

Эскалация без ожидания обязательна, если:
- продажа невозможна на нескольких устройствах одновременно;
- очередь синка растёт и не убывает > 15 минут;
- есть массовый отказ лицензии при доступном backend;
- отчёты отдают пустые/неконсистентные данные на всех терминалах.

## Минимальный пакет артефактов для инцидента

1. Скриншот экрана с симптомом.
2. Время инцидента и timezone.
3. Device ID / версия приложения.
4. Запрос/ответ проблемного endpoint (если есть).
5. Статус синка (сколько чеков `PENDING_SYNC` / `ERROR` / `SYNCED`).

Этого достаточно, чтобы backend/android команда воспроизвела и локализовала проблему без повторного запроса к кассиру.
