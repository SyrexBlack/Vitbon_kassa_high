# Vitbon ККМ — Архитектурное проектирование

**Версия:** 1.0
**Дата:** 2026-04-08
**Статус:** черновик

---

## 1. Цель и назначение

Полноценное Android-приложение кассового аппарата (ККМ/ККТ) для российской розницы,
работающее на **любом Android-устройстве** (Android 7+). Соответствует 54-ФЗ, интегрировано
с ЕГАИС (полный цикл), «Честным Знаком», ОФД. Собственный бэкенд на Kotlin/Spring Boot.
Масштаб — федеральная сеть 50+ касс с критичным офлайн-режимом.

---

## 2. Ограничения и допущения

- Каждое Android-устройство работает **автономно** при потере связи до 30 дней (54-ФЗ)
- Универсальная поддержка оборудования через абстракцию `DeviceBridge` (плагины-драйверы)
- ГОСТ-криптография: BouncyCastle + JCE-Provider (либо BKS-хранилище)
- Короткий цикл MVP: все интеграции сразу в фазе 1
- Tech stack: Kotlin 1.9+, Jetpack Compose, Spring Boot 3, PostgreSQL, Redis Streams

### 3.2. Принцип гибридной синхронизации

Каждый терминал — **автономный узел** с локальной SQLite (чеки, товары, конфигурация).
Бэкенд — централизованный источник истины. Синхронизация — pull/push через REST + Redis.

**Почему не тонкий клиент:** при 50+ кассах и 30-дневном окне офлайна терминалы
должны работать полностью автономно. "Толстый клиент" с полной локальной базой —
единственный способ обеспечить это.

**Почему не полностью автономный:** нужен централизованный каталог, управление правами,
консолидированная аналитика, обновление цен сети.

### 3.3. Стек

| Слой | Технологии |

### 4.2. Синхронизация (WorkManager)

- **Sync Down** (pull): каталог товаров, конфигурация, справочники ЕГАИС/ЧЗ
- **Sync Up** (push): чеки, акты ЕГАИС, уведомления ЧЗ
- Интервал: каждые 30 сек при наличии сети; сразу при восстановлении связи
- Долговечность: WorkManager гарантирует выполнение даже при закрытии приложения
- Конфликты: merge по timestamp (сервер — последний выигрывает), device_id для идентификации источника
- Офлайн-буфер: до 10 000 чеков в Room (зашифровано SQLCipher)

### 4.3. Фискальный регистратор (DeviceBridge)

```kotlin
interface FiscalDevice {
    suspend fun openShift(): FiscalResult
    suspend fun printCheck(check: FiscalCheck): FiscalResult
    suspend fun closeShift(): FiscalResult
    suspend fun getStatus(): DeviceStatus
}
```

Драйверы загружаются динамически по модели устройства (vendor/product ID или ручной выбор).
Список начально поддерживаемых моделей: АТОЛ 30Ф/55Ф/90Ф, Штрих-М, Dreamkas Ф,
Нева 01-Ф (встроенный ФН).

### 4.4. Безопасность Android

- Приложение **не работает на root-устройствах** (проверка su binary, SafetyNet/Play Integrity)
- Приложение **не работает при выключенном SELinux** (проверка `enforcing`)
- Ключи (ЧЗ, ЕГАИС) — в Android Keystore / hardware-backed KeyMaster
- TLS mutual authentication: клиентский сертификат в KeyStore
- SQLCipher-шифрование локальной БД (пароль из Keystore)

## 5. Backend

### 5.1. Endpoints (REST API)

| Метод | Путь | Описание |
|-------|------|---------|
| POST | `/api/v1/checks` | Создать/отправить чек |
| GET | `/api/v1/checks/{id}` | Статус чека |
| GET | `/api/v1/checks/sync` | Синхронизация чеков (down) |
| GET | `/api/v1/products` | Каталог товаров |
| POST | `/api/v1/products/sync` | Синхронизация товаров (down) |
| POST | `/api/v1/auth/login` | Авторизация кассира |
| GET | `/api/v1/shifts/{cashierId}` | Отчёты по смене |
| POST | `/api/v1/egais/incoming` | Принять накладную ЕГАИС (прокси) |
| POST | `/api/v1/chaseznak/sell` | Продажа маркированного (прокси) |

### 5.2. Очередь фискализации (Redis Streams)

```
XADD fiscal-queue "*" checkId <uuid> payload <json>
```

Worker читает очередь → формирует ФД → отправляет в ОФД → обновляет статус чека.
При ошибке — retry с exponential backoff (макс. 5 попыток), затем переводит в ERROR
и уведомляет кассу.

### 5.3. Масштабирование (50+ касс)

- **Горизонтальное**: 2+ API-сервера за балансировщиком (HAProxy/Nginx)
- **Очередь**: Redis Streams — хватает для 500+ чеков/мин
- **БД**: PostgreSQL master-slave, пул соединений (HikariCP, макс 50 соединений на инстанс)
- **ОФД-лимит**: не более 20 параллельных запросов к одному ОФД (ограничение оператора)
- **Хранилище**: S3/NAS — PDF-архивы чеков (~1 MB/чек, терабайты/год)

### 5.4. Мониторинг

- Метрики: latency API, длина очереди, время обработки чека, ошибки ОФД/ЕГАИС/ЧЗ
- Alerting: Telegram/SMS при 3+ последовательных ошибках фискализации
- Логи: структурированные (JSON), ELK-совместимые

## 6. Интеграции

### 6.1. ОФД

- Формат: ФФД 1.2 (теги 1125, 1187, 1008, 1234–1238)
- Протокол: REST (JSON) — большинство ОФД; SOAP — Такском
- Поток: чек → Worker → ОФД → ФНС автоматически
- Офлайн: при недоступности ОФД чеки накапливаются, синхронизируются при восстановлении
  (окно 30 дней по 54-ФЗ)

### 6.2. ЕГАИС (УТМ)

- Полный цикл: приём накладных, акты вскрытия тары, списание, инвентаризация
- УТМ запущен **на стороне бэкенда** (сервер в ЦОД) — не на каждой кассе
- API УТМ: REST + RSA-подпись (запросы через HTTP к localhost:8080 УТМ)
- Сертификаты: RSA-ключ магазина в УТМ, обновление раз в 1–2 года

### 6.3. Честный Знак (ЧЗ)

- ЛМ ЧЗ запущен **на стороне бэкенда**
- API: REST + КЭП ГОСТ (сертификат в KeyStore сервера)
- Поток: сканирование 2D-кода → запрос в ЛМ ЧЗ → проверка → списание
- Отправка уведомления о реализации (теги 1162/1163 в ФФД)

### 6.4. Эквайринг + СБП

- Эквайринг: SDK банка-эквайера (AIDL/PINPad SDK)
- СБП: API банка или сервис-провайдер (Плати QR, СберQR)
- Оплата split: часть наличными, часть картой — корректно разбивается по тегам 1234–1238

## 10. Безопасность

- **Сервер**: TLS 1.3, mutual auth, шифрование БД (pgcrypto), бэкапы ежедневные
- **Android**: Keystore, SQLCipher, no root, SELinux enforcing
- **Ключи**: УТМ-сертификат — на сервере (не в кассе), ЧЗ КЭП — в HSM/KeyStore сервера
- **Соответствие**: 152-ФЗ (ПДн), 275-ФЗ (гостауентичность), 54-ФЗ (ККТ)

## 11. Риски

| Риск | Вероятность | Мера |
|------|------------|------|
| Изменение формата ФФД | Высокая | Модульная архитектура, версионирование ФД |
| Утечка ключей ЕГАИС/ЧЗ | Низкая | HSM, аудит-логи, ограничение доступа |
| Несовместимость драйвера ФР | Средняя | Plugin-модель, абстракция, ручной fallback |
| Превышение окна офлайна 30 дней | Низкая | Мониторинг, push-уведомление при >20 дней |
| Блокировка ОФД/ЕГАИС/ЧЗ | Средняя | Graceful degradation, офлайн-буфер |
