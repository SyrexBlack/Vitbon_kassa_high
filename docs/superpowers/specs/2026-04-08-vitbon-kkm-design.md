# Vitbon ККМ — Архитектурное проектирование

**Версия:** 2.0
**Дата:** 2026-04-08
**Статус:** черновик

---

## 1. Цель и назначение

Мобильное кассовое приложение «Мобильная касса VITBON» для Android 6.0+,
соответствующее 54-ФЗ. Обеспечивает базовый функционал кассира, интеграцию
с ККТ MSPOS-K и Нева 01Ф, облачный товароучёт, все способы оплаты.
Опциональные модули маркировки (Честный ЗНАК) и ЕГАИС подключаются
за дополнительную плату. Масштаб — массовое внедрение, 200+ касс.

---

## 2. Ограничения и допущения

- ККТ: **только** MSPOS-K и Нева 01Ф (фискальное ядро предоставляется производителем)
- Android 6.0 (API 23) и выше
- Офлайн-критичность: фискальные чеки работают без интернета; облачная синхронизация — с буферизацией
- ЕГАИС и ЧЗ — **отключаемые модули**, активируемые по ключу/флагу
- Лицензирование: проверка через API сервера лицензий; grace period 7 дней
- 54-ФЗ: смена более 24 ч. → предложение закрыть; закрытая смена → автооткрытие перед первым чеком
- ФФД: версия определяется от ФН; переключение — только при замене ФН

## 3. Архитектура системы

### 3.1. Высокоуровневая схема

```
┌──────────────────────────────────────────────────────────────┐
│  Android-терминал (каждая касса)                            │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────────┐   │
│  │ Kotlin/      │  │ Room/SQLite │  │  FiscalCore SDK   │   │
│  │ Compose UI   │  │ (offline)   │  │  (MSPOS-K / Нева) │   │
│  └──────┬───────┘  └──────┬───────┘  └────────┬─────────┘   │
│         │                 │                    │              │
│   ┌─────┴─────────────────┴────────────────────┴──────┐    │
│   │        SyncWorker (WorkManager)                    │    │
│   │   (буферизация чеков, синхронизация товаров)       │    │
│   └────────────────────────┬───────────────────────────┘    │
└─────────────────────────────┼────────────────────────────────┘
                              │  HTTPS/REST
              ┌───────────────┼───────────────┐
              │               │               │
        ┌─────┴─────┐  ┌──────┴──────┐  ┌─────┴──────┐
        │ Backend   │  │ Лицензий    │  │ УТМ ЕГАИС  │
        │ (Kotlin/  │  │ Сервер      │  │ (optional) │
        │ Spring    │  │             │  └────────────┘
        │ Boot)     │  └─────────────┘
        │           │                   ┌────────────┐
        │           │                   │ ЛМ ЧЗ      │
        │           │                   │ (optional)│
        └─────┬──────┘                   └────────────┘
              │
        ┌─────┴──────┐  ┌────────────┐  ┌────────────┐
        │ ОФД       │  │ Облачный   │  │ Цифровой  │
        │           │  │ товароучёт │  │ ID Max    │
        └───────────┘  └────────────┘  └────────────┘
```

### 3.2. Гибридная синхронизация

Каждый терминал — автономный узел (Room/SQLite). Бэкенд — центр.
Синхронизация: push чеков → pull товаров. При 200+ кассах — пакетная
синхронизация по расписанию (каждые 30 сек) +增量 sync при изменении.

### 3.3. Стек

| Слой | Технологии |
|------|-----------|
| **Android App** | Kotlin 1.9, Jetpack Compose, Room + SQLCipher, WorkManager, Hilt, Coroutines |
| **ККТ SDK** | MSPOS-K SDK, Нева 01Ф SDK (провайдерские библиотеки) |
| **Backend** | Kotlin + Spring Boot 3, Coroutines, JPA/Hibernate, Flyway |
| **БД Backend** | PostgreSQL 16+ (master-slave), Redis Streams |
| **Лицензирование** | Собственный сервер лицензий (REST API) |
| **Infra** | Docker/Kubernetes, S3 (архивы чеков) |

### 4.2. Фискальное ядро (адаптер)

```kotlin
interface FiscalCore {
    suspend fun openShift(): FiscalResult
    suspend fun printSale(check: FiscalCheck): FiscalResult
    suspend fun printReturn(check: FiscalCheck): FiscalResult
    suspend fun printCorrection(doc: CorrectionDoc): FiscalResult
    suspend fun closeShift(): FiscalResult
    suspend fun printXReport(): FiscalResult
    suspend fun cashIn(amount: Money, comment: String?): FiscalResult
    suspend fun cashOut(amount: Money, comment: String?): FiscalResult
    suspend fun getStatus(): FiscalStatus   // ФН, смена, номер чека
    suspend fun getFFDVerson(): FFDVersion // "1.05" или "1.2"
}
```

Реализации: `MSPOSKFiscalCore`, `Neva01FFiscalCore`.
Выбор — по модели устройства (DeviceInfo) или ручной настройке.

### 4.3. Синхронизация (WorkManager)

- **Sync Up**: чеки → бэкенд (буферизация в Room, повтор до подтверждения)
- **Sync Down**: товары, цены, остатки, конфигурация
- **Интервал**: 30 сек при наличии сети; сразу при восстановлении
- **Конфликты**: server wins по timestamp

### 4.4. ФФД 1.05 / 1.2

```
Алгоритм:
  1. Старт → проверка смены (открыта / закрыта / >24ч)
  2. >24ч → предложение закрыть смену
  3. Закрыта → открыть (авто или ручная команда)
  4. Перед каждым чеком → getFFDVersion() → выбрать набор тегов
  5. При ошибке "неверный формат" → повторный getFFDVersion() → перестроить
```

Переключение версии ФФД — только при замене ФН (не в приложении).

### 4.5. Безопасность Android

- Проверка root — блокировка запуска
- SELinux enforcing
- Ключи — Android Keystore / hardware-backed KeyMaster
- SQLCipher-шифрование Room (пароль из Keystore)
- TLS mutual authentication

### 4.6. Лицензирование

```
Старт → /api/license/check → { status: ACTIVE | EXPIRED }
  ACTIVE → работа
  EXPIRED → блокировка (кроме отчётов и настроек) + сообщение
  Нет сети → grace period 7 дней (счётчик в SharedPreferences)
  Каждые 24ч → повторная проверка
```

## 5. Мониторинг статусов

| Статус | Состояния | Действие при проблеме |
|--------|-----------|----------------------|
| Интернет | Есть / нет | Блокировка синхронизации; фискальные чеки работают |
| Облачный сервер | Доступен / ошибка | Время последней синхр., кнопка повтора |
| ОФД | Подключено / задержка | Кол-во непереданных чеков |
| ЛМ ЧЗ *(модуль активен)* | Установлен / доступен | Предложение установить, блокировка маркировки |
| УТМ ЕГАИС *(модуль активен)* | Установлен / доступен | Блокировка продажи алкоголя |
| Статус тарифа | Активен / просрочен / нет связи | Блокировка при просрочке (grace period 7 дней) |

## 6. Опциональные модули (feature flags)

### 6.1. Модуль маркировки (Честный ЗНАК)

```
Активация: remote key / флаг в конфигурации (без переустановки)
  - Сканирование DataMatrix → валидация через ЛМ ЧЗ или облачный запрос
  - Выбытие при продаже (групповое)
  - Приёмка / возврат маркированного
  - Блокировка продажи при ошибке валидации
```

### 6.2. Модуль ЕГАИС

```
Активация: remote key / флаг (без переустановки)
  - Интеграция с УТМ (настраиваемый адрес)
  - Фиксация продажи алкоголя (incl. пиво в кегах)
  - Акт вскрытия тары
  - Цифровой ID Max: сканирование QR → API Max → проверка возраста
    → при неподтверждении блокировка + логирование
```

### 7.2. Масштабирование (200+ касс)

- **API**: 3+ инстанса за балансировщиком (HAProxy)
- **Очередь**: Redis Streams — пакетная обработка чеков (batch insert)
- **БД**: PostgreSQL, read replica для чтения отчётов
- **Хранилище**: S3 — PDF архивы чеков
- **ОФД**: не более 20 параллельных запросов (ограничение оператора)

## 10. Риски

| Риск | Вероятность | Мера |
|------|------------|------|
| Изменение формата ФФД | Высокая | Модульное формирование ФД по версии |
| Изменение API производителя ККТ | Средняя | Адаптер скрывает детали SDK |
| 200+ касс — конфликты синхронизации | Низкая | Batch sync, server-wins merge |
| Root-устройство у кассира | Средняя | Блокировка запуска |
| Просрочка лицензии + отсутствие сети | Низкая | 7-дневный grace period |
